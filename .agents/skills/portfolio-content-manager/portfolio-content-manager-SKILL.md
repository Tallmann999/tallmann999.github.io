# Portfolio Content Manager Skill

## Цель

Помогай дорабатывать статическое портфолио Unity / C# разработчика на GitHub Pages.

Основная задача: аккуратно добавлять новые блоки, проекты, ссылки, контакты, описания, gif/video/screenshots и улучшать страницу без поломки текущего дизайна.

## Когда использовать skill

Используй этот skill, если пользователь просит:

- добавить новый проект в портфолио;
- заменить placeholder проекта на реальный проект;
- добавить gif, видео, скриншот или ссылку;
- обновить блок "Обо мне";
- обновить навыки;
- добавить Telegram, LinkedIn, Itch.io, Reddit, GitHub, email;
- улучшить текст для портфолио;
- сделать страницу более профессиональной для Unity / C# вакансий;
- добавить новый раздел: Devlog, Tools, AI workflow, Game Design, Case Study, Contacts;
- проверить портфолио перед публикацией.

## Структура проекта

Ожидаемая структура:

```text
/
├─ index.html      # Основная страница
├─ styles.css      # Стили
├─ script.js       # Небольшая интерактивность
├─ assets/         # Изображения, gif, видео, превью проектов
└─ README.md       # Описание репозитория
```

Если структура отличается, сначала изучи реальные файлы и адаптируй действия.

## Главные правила

1. Не переписывай весь сайт без необходимости.
2. Вноси маленькие, понятные изменения.
3. Сохраняй текущий стиль: темный фон, карточки, акцентные цвета, аккуратная сетка.
4. Не добавляй React, Vite, Tailwind, Bootstrap или другие фреймворки, если пользователь прямо не попросил.
5. Не ломай GitHub Pages: сайт должен открываться как обычный статический HTML.
6. Не выдумывай ссылки, контакты, даты, названия проектов и технологии.
7. Если данных не хватает, используй понятный placeholder и пометь, что нужно заполнить.
8. Все пользовательские тексты делай профессиональными, но простыми.
9. Для портфолио Unity / C# делай акцент на механиках, архитектуре, геймдизайне и результате.
10. После изменений кратко объясни, какие файлы изменены и что проверить.

## Как добавлять проект

Когда пользователь дает новый проект, добавь или обнови карточку в разделе "Портфолио".

Стандартная структура проекта:

- Название проекта
- Короткий pitch в 1 предложении
- Описание 2-4 строки
- Роль пользователя
- Что реализовано
- Технологии
- Ссылки: GitHub, Demo, Itch.io, видео
- Медиа: screenshot/gif/video из `assets/`

Если проект еще сырой, показывай его как prototype / work in progress, но профессионально.

## Шаблон карточки проекта

Используй похожую HTML-структуру, адаптируя под текущий `index.html`:

```html
<article class="project-card reveal">
  <div class="media-placeholder">
    GIF / Video placeholder
  </div>
  <div class="project-body">
    <h3>Название проекта</h3>
    <p>Краткое описание проекта: жанр, core mechanic, что игрок делает.</p>
    <ul class="tech-list">
      <li>Unity</li>
      <li>C#</li>
      <li>Game Design</li>
    </ul>
    <div class="project-links">
      <a href="#" aria-label="Открыть GitHub проекта">GitHub</a>
      <a href="#" aria-label="Открыть демо проекта">Demo</a>
    </div>
  </div>
</article>
```

Если есть реальное изображение или gif, замени `media-placeholder` на:

```html
<img class="project-media" src="assets/project-name-preview.gif" alt="Gameplay preview проекта Название проекта">
```

И добавь CSS для `.project-media`, если его еще нет:

```css
.project-media {
  width: 100%;
  min-height: 184px;
  display: block;
  object-fit: cover;
  border-bottom: 1px solid var(--color-border);
  background: #111827;
}
```

## Как писать описание проекта

Для каждого проекта желательно использовать формулу:

```text
[Жанр] / [платформа] / [главная механика].
Я отвечал за [системы/код/дизайн].
Главный фокус: [что показывает навык].
```

Пример:

```text
2D action roguelite prototype на Unity с системой цветных печатей и элементального оружия. Я реализовал базовую боевую механику, логику врагов, модульную стрельбу и игровые состояния. Проект показывает работу с C#, компонентной архитектурой и быстрым прототипированием.
```

## Приоритетные блоки портфолио

Если пользователь просит улучшить портфолио, предлагай и добавляй блоки в таком порядке:

1. Hero block: кто я и чем полезен.
2. Featured Projects: 2-3 сильных проекта.
3. Case Study: один подробный разбор проекта.
4. Skills: Unity, C#, Gameplay, UI, AI, Tools, Git.
5. Devlog / Process: как пользователь работает.
6. Contacts: GitHub, Telegram, LinkedIn, Itch.io, email.

## Что важно для Unity / C# портфолио

Подсвечивай:

- Gameplay systems
- Enemy AI
- Weapon systems
- UI / UX
- Inventory / items
- Save/load
- Tools for development
- SOLID / clean architecture
- ScriptableObject configs
- Object pooling
- State machines
- Git workflow
- Performance / optimization
- WebGL / itch.io builds

## Контроль качества перед завершением

Перед финальным ответом проверь:

- все ссылки имеют понятный текст;
- нет битых закрывающих тегов;
- изображения имеют `alt`;
- навигация не сломана;
- карточки проектов выглядят одинаково;
- адаптивность не сломана;
- нет лишних зависимостей;
- текст не выглядит как "заглушка", если пользователь дал реальные данные;
- если добавлены новые assets, пути совпадают с реальными именами файлов.

## Формат ответа пользователю

В конце работы напиши:

1. Что изменено.
2. Какие файлы изменены.
3. Что пользователю нужно проверить в браузере.
4. Какие данные еще стоит добавить позже.

Не пиши длинные теоретические объяснения, если пользователь просил конкретное изменение.
