Project Overview

Caliber of Chaos is a 2D action shooter prototype where enemies are protected by elemental seals. The player must use the correct weapon element to break seals before dealing real HP damage.

The main design goal is to combine bullet-hell action with puzzle-like combat decisions.

My Role

Solo Unity / C# Developer. I designed and implemented gameplay systems, enemy behavior, elemental combat, mutation mechanics, UI feedback, and playable WebGL demo flow.

Core Gameplay Loop

  1. Enter a corrupted area.
  2. Identify enemy seals and mutation types.
  3. Use the correct elemental weapon.
  4. Break seals in the correct order.
  5. Damage enemies after protection is removed.
  6. Clean corrupted objects and transform some of them into allies.
  7. Survive waves and move to the next encounter.

Implemented Systems

Elemental Combat System

The combat system is based on several element types: Bio Red, Crystal Blue, Purge Green, and Gold Yellow. Enemies can react differently depending on their current mutation and active seal.

Seal System

Some enemies and objects have elemental seals. A normal attack cannot damage them until the correct seal is broken. This creates a combat-puzzle layer on top of the shooter gameplay.

Enemy Behavior

The prototype includes melee enemies, shooter enemies, patrol enemies, elite enemies, and spawner-based enemy logic.

Mutation / Biome System

Mutation zones affect the ground, enemies, and objects. Purge Green can clean corrupted objects, while other elements can create danger zones or change enemy behavior.

Environmental Allies

Some objects can be transformed into temporary allies. For example, trees can become friendly units and help the player fight enemies for a limited time.

Technical Architecture

The project is built with small Unity components instead of one large controller. The goal is to keep gameplay logic readable, modular, and easier to extend. 

Player
 ├── PlayerMovement
 ├── PlayerWeapon
 ├── InteractionHandler
 └── Inventory

Weapon
 ├── WeaponFrame
 ├── WeaponCore
 ├── WeaponMod
 └── BulletSpawner

Enemy
 ├── EnemyMovement
 ├── EnemyBehaviour
 ├── Health
 ├── MutationController
 └── SpellNodesCounter

Bullet
 ├── BulletMover
 ├── BulletView
 ├── BulletContext
 └── Impact Processing

Technical Challenge

Problem

Bullet collision logic became too large. It started handling damage, seals, mutation, faction rules, object interactions, explosions, and pool return logic.

Solution

I started separating bullet behavior into smaller responsibilities: movement, visuals, collision processing, damage rules, seal interaction, and pool return logic. This makes the system easier to extend with new elemental effects.
